### 微信机器人概念设计

**用户需求：** "我如果做一个微信机器人出来，我发产品的名称年份或者 OE 号给机器人，就能马上网上找到对应的产品图片"

1. 用户通过微信发送产品名称/年份/OE号

## 技术博客：浏览器自动化的"身份困境"

### 问题

AI 需要操作浏览器，但有两种完全不同的方式：

| 方式 | 优点 | 缺点 |
|------|------|------|
| 独立启动 Puppeteer | 完全可控 | 无登录态、被网站检测 |
| CDP 连接已有浏览器 | 有登录态 | 需要用户手动启动、调试端口 |

### 用户真正的需求

用户要的不是"一个能操作浏览器的 AI"，而是**"一个能用我已经登录好的浏览器去做事的 AI"**。

这个区别至关重要：
- 用户已经有浏览器窗口（8 个标签页，包含 Facebook、Instagram、外贸通等）
- 所有网站都已经登录
- 用户只想要 AI 去**驱动这些已有的窗口**

### Puppeteer MCP 配置

```json
{
  "mcpServers": {
    "puppeteer": {
      "command": "npx",
      "args": [
        "-y",
        "@anthropic-ai/claude-code-mcp-puppeteer",
        "--port",
        "18800"
      ]
    }
  }
}
```

### 关键：CDP 连接的正确方式

```javascript
// 错误的：每次启动新浏览器
const browser = await puppeteer.launch(); // ❌ 无登录态

// 正确的：连接已有浏览器
const browser = await puppeteer.connect({
  browserURL: 'http://127.0.0.1:18800',
  defaultViewport: null
}); // ✅ 有登录态

// 查看已有页面
const pages = await browser.pages();
const page = pages[0]; // 使用第一个标签页
```

## 知识库

### 微信机器人架构设想

```
用户 → 微信消息 → 微信机器人 → Webhook → 后端服务
  → 搜索产品图片（Google/eBay/RockAuto/Alibaba）
  → 交叉核验 → 返回图片给用户
```

### ngrok 配置

```powershell
# 安装 ngrok
winget install ngrok

# 配置认证密钥
ngrok config add-authtoken <your-token>

# 创建隧道（转发本地端口到公网）
ngrok http 3000

# 后台运行
start-process -NoNewWindow ngrok -ArgumentList "http 3000"
```

### 浏览器自动化方案演进

```
V1: 独立启动 Chrome（无登录态） ❌
V2: CDP 连接已有 Chrome（依赖手动启动） ⚠️
V3: Puppeteer MCP + CDP 持久连接 ✅
```

## 复盘

**对的事：**
- 微信机器人概念很有价值——输入产品信息就能获得对应图片，外贸业务场景极其实用
- 最终安装了 Puppeteer MCP，为后续浏览器自动化打下基础
- ngrok 配置完成，解决了内网穿透问题

**不对的事：**
- **浏览器自动化绕了太多弯路。** 用户从 5/14 就开始强调"用真实浏览器"，但直到 5/18 才真正安装 Puppeteer MCP。这个问题花了 5 天来解决，但实际上第一天就该解决
- 用户说"你这个逻辑很乱"、"你能先把这些浪费时间的东西优化好先吗" —— 这是对我最大的批评。浏览器自动化的问题我一直在用临时方案补丁，没有从根本上解决
- 企业微信 Bot 尝试了很久才发现权限限制严格，走不通。应该先了解企业微信的权限模型再动手

**根本原因：** 我倾向于用代码临时解决问题，而不是找成熟的工具/方案。用户一直在说"你去网站看看有什么技能包"，意思是让我去找现成的 MCP 工具，而不是自己写脚本。

**今天最大的教训：** 当用户反复说同一个问题时，说明这个问题是真正的瓶颈，必须从根本上解决，不能用补丁凑合。
