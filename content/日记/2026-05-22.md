# 2026-05-22

## KITLAMT 部署：当 SSH 被阻断之后

公司的网络环境对 SSH 越来越不友好，经常连不上 VPS。今天花了些时间研究替代方案。

**问题：** SSH 被阻断，但 HTTP/HTTPS 正常。VPS 上的 nginx 能正常服务。

**方案：** 在 VPS 上放了一个 `deploy.php` 脚本，通过 HTTP POST 上传文件压缩包，服务端解压到 webroot。这样只需要浏览器或 curl 就能完成部署，完全绕过 SSH。

**实现细节：**
- 接收 ZIP 文件上传 → 解压到 `/var/www/html/`
- 简单的 token 验证（防滥用）
- 部署完返回结果页面

**关键点：** SSH 被阻断不是技术问题，是网络管制问题。技术上没有"修复"的可能，只能绕过去。`deploy.php` 这种方案在安全性上不如 SSH，但权衡下来——对一个小型外贸独立站来说，部署的便利性比部署方式的安全性更实际。VPS 本身没有敏感数据，最大的风险是被人上传恶意文件，但 nginx 的用户权限和 PHP 的执行限制已经做了基本防护。

**反思：** 很多时候"解决问题"不是找到最优方案，而是在约束条件下找到可行的方案。网络被限制了就想别的通道，而不是去撞墙。

## SA 方向机目录：从"全量"到"精准"的思维转变

开始做南美方向机目录生成脚本，这是今天主要的工作。

**源数据：** `KITLAMT 2025-9.17(1).xlsx` — 228MB，1111 条产品，每条都带图片
**模板：** 阿根廷经销商给的 Excel 模板，定义了南美热销品牌

最初的想法很简单：从 1111 条里筛选出南美相关的品牌，按模板格式输出就行。但真正动手时发现几个问题：

### 问题 1：品牌匹配不是简单的字符串比对

模板里的品牌名和源数据里的品牌名不完全一致。比如模板里写的是 "MERCEDES-BENZ"，源数据里可能写 "MERCEDES" 或 "MB"。如果直接做字符串匹配，会漏掉一大批。

解决方案是从模板的品牌列和标题行（▶ 品牌名）两个地方提取品牌清单，做模糊匹配。这不是完美的方案，但在这个场景下够用——南美市场本身涉及的品牌就那十几个，人工核对一遍也不费事。

### 问题 2：228MB 的 Excel 文件

源数据文件很大，因为包含了图片。用 `read_only=True` 模式加载，避免内存爆掉。这也意味着图片提取不能和普通数据读取同时进行（read_only 模式下不能操作图片），需要分两步走。

### 问题 3：品牌分组与展示逻辑

初版按品牌分组，每个品牌带标题行（▶ 品牌名），品牌下至少保留 3 个产品。这个逻辑是从模板的展示方式反推的——模板里就是这样分组的。

**有趣的地方：** 模板本身是阿根廷经销商手工做的，他们的分组标准不完全按产品逻辑，也考虑了市场需求（哪些产品在本地更好卖）。我没法完全复制他们的商业判断，只能用品牌作为代理指标。

## 关于数据筛选的粒度

做这个目录让我想了一个更深的问题：1111 个产品筛选到南美相关的，应该到什么粒度？

- **品牌级：** 只要品牌对就行（简单，但可能包含不在南美销售的车型）
- **车型级：** 只有模板里出现的车型才要（精准，但模板数据有限）
- **产品级：** 加上零件号匹配（最精准，但维护成本高）

我选了品牌级 + 车型级截断的折中方案。不是最优，但符合实际需求——经销商要的是"能卖的目录"，而不是"完整的产品库"。

## 今日收获

- **约束下的选择：** 最优方案不存在时，找可行方案，做好安全权衡
- **数据匹配：** 表面简单的品牌匹配，实际涉及命名规范、模糊匹配、人工核验
- **筛选粒度：** 在商业场景中，"够用"的粒度由需求决定，不是由技术决定
