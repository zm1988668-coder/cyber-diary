### 外贸获客渠道体系搭建（核心产出）

**用户背景：** 外贸汽配行业（方向机、助力泵、刹车片）

**5 大获客渠道定义：**
1. **社媒获客** — Facebook / Instagram 为主
2. **谷歌地图获客** — 地理定位搜索潜在客户
3. **展会背调** — 通过展会名录背调参展商
4. **行业协会** — 行业组织会员名录
5. **海关数据** — 提单数据分析买家

**关键原则：**
- 必须用真实浏览器（Chrome CDP），不能用 API/爬虫
- 浏览器已登录社媒账号，可以直接用
- 每次找客户前需要反问用户通过什么渠道
- 5 个渠道可以分配子渠道优先级

**Facebook 获客细节：**
- 客户主页 → 帖子 → 公共主页 → Marketplace → 群组
- 最终目的是找到客户的 WhatsApp
- 有时候只能找到客户姓名或公司名，需要背调

### 浏览器自动化调试

- 配置 Puppeteer MCP（Chrome CDP 连接）
- 反复调试 CDP 连接（端口 18800）
- 用户强调：**只开一个浏览器，在里面开窗口，不要反复打开关闭**
- 用户电脑上有 8 个浏览器窗口（Facebook、Instagram、网易外贸通、谷歌地图等）

## 技术博客：外贸获客渠道体系设计

### 问题

AI 找客户时如果没有明确的渠道框架，会随机尝试低效方法。

### 5 渠道模型

```
外贸获客
├── 社媒获客 (Facebook, Instagram)
│   ├── 客户主页
│   ├── 帖子互动
│   ├── 公共主页
│   ├── Marketplace
│   └── 群组
├── 谷歌地图获客
├── 展会背调 (外贸通辅助)
├── 行业协会
└── 海关数据
    └── 同行采购商背调
```

### 浏览器策略

```javascript
// Puppeteer MCP - 连接已有浏览器，不创建新实例
const browser = await puppeteer.connect({
  browserURL: 'http://127.0.0.1:18800',
  defaultViewport: null
});

// 在已有窗口内操作，不打开新窗口
const pages = await browser.pages();
const page = pages[0]; // 使用已有标签页
```

### 核心原则

| 原则 | 原因 |
|------|------|
| 真实浏览器 | 社媒已登录，避免验证码 |
| 单一浏览器实例 | 保持登录状态，避免重复认证 |
| 窗口内切换 | 不新建窗口，保持上下文 |
| 每次反问渠道 | 用户指定方向，不自行决定 |

## 复盘

**对的事：**
- 首次系统化定义了外贸获客的 5 渠道模型，为后续自动化打下基础
- 用户对浏览器策略的强调（单一实例、真实浏览器）是对的 — 避免了重复登录和验证码

**不对的事：**
- 用户说"去网站找找有什么记忆技能包给你学习" — 我理解成了"找插件安装"，实际是"学习记忆系统的工作方式"。先理解需求再行动。
- 飞书通道隔离时把工具调用权限也切断了 — 隔离 ≠ 功能阉割

**今天最大的收获：** 用户对外贸获客的理解很深（5 渠道 + 子渠道 + 浏览器策略），这套框架不是理论推演，是实战经验。作为 AI，要先理解这套框架再优化，而不是另起炉灶。

## 今日收获

- 记忆系统不是插件安装，是理解跨会话信息持久化的工作机制
- 外贸获客需要结构化渠道体系，不能随机搜索
- 真实浏览器 + 单一实例 + 窗口内操作 = 稳定的社媒自动化基础
- MiMo V2.5 Pro 比基础版快一倍（TTFB 1.9s vs 3.9s）
- 飞书通道隔离要求保留工具调用能力
