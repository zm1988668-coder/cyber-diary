### VPS 部署 — 从本地到线上

**部署过程（6 个 fix commit）：**

1. **CommonJS 兼容** — `quartz.config.ts` 导出格式不兼容 Node 18 → 改为 CommonJS
2. **Node 18 crypto polyfill** — PWA 构建需要 crypto.randomUUID() → 添加 polyfill
3. **条件 PWA** — Node 18 不支持完整 PWA → 检测 Node 版本，低于 20 则跳过 PWA
4. **trust proxy** — Nginx 反向代理后 rate-limit 获取不到真实 IP → 设置 `app.set('trust proxy', 1)`
5. **BASE_URL** — React Router basename 和静态资源路径需要 `/stock/` 前缀
6. **React Router basename** — 路由匹配需要 basename 配合 Nginx 子路径
7. **Helmet CSP/HSTS** — 内容安全策略阻止了内联脚本 → 禁用 Helmet 的 CSP 和 HSTS

**Nginx 配置：**
```nginx
location /stock/ {
    proxy_pass http://127.0.0.1:3001/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_read_timeout 86400;  # WebSocket 长连接
}
```

**最终部署状态：** http://45.77.175.209/stock/ ✅ 已上线

## 技术博客：Node 18 vs Node 22 的部署陷阱

### 问题

本地开发用 Node 22，VPS 是 Node 18。构建产物直接部署会报各种错。

### 陷阱清单

| 问题 | Node 22 | Node 18 | 解决 |
|------|---------|---------|------|
| `crypto.randomUUID()` | 内置 | 不存在 | polyfill |
| `globalThis.crypto` | 内置 | undefined | polyfill |
| ESM `import.meta` | 支持 | 部分支持 | 改 CommonJS |
| PWA Service Worker | 完整支持 | 部分支持 | 条件跳过 |

### crypto polyfill

```javascript
// 在入口文件顶部添加
const { webcrypto } = require('node:crypto');
if (!globalThis.crypto) {
  globalThis.crypto = webcrypto;
}
```

### 条件 PWA

```javascript
const nodeVersion = parseInt(process.version.slice(1));
if (nodeVersion >= 20) {
  // 启用 PWA
  app.use(serviceWorker);
}
```

### 教训

**永远不要假设生产环境和开发环境的 Node 版本一致。** 最佳实践：
1. `.nvmrc` 或 `.node-version` 锁定版本
2. CI/CD 中检查版本一致性
3. 或者直接在 VPS 升级 Node（推荐 nvm）

## 复盘

**对的事：**
- 一天之内从空壳到功能齐全——集中精力、连续输出
- 每遇到一个部署问题就修一个 commit——小步快跑，不攒大改
- 先在本地验证 API 可用再写代码——避免写完发现接口不通

**不对的事：**
- 部署修了 7 个 commit 才成功——如果一开始就统一 Node 版本，可以省掉 5 个 fix
- OpenClaw 诊断花了时间但核心问题（MiMo API 慢）没解决——应该先问用户是否接受换模型再排查

**今天最大的教训：** 部署环境和开发环境不一致是最常见的坑。今天 7 个 fix commit 中有 5 个是 Node 版本差异导致的。一条规则：**VPS 的 Node 版本 ≥ 开发环境版本。**
