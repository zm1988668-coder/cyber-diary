## 技术博客：Multi-Agent 工作流的 Token 消耗模型

### 问题

单次对话的 Token 消耗容易估算，但 Multi-Agent 并行工作流的消耗往往被低估。

### 消耗模型

```
传统模式（1 Agent）：
用户 → AI → 完成
Token = 1 × prompt + completion

Multi-Agent 模式（5 Agents）：
PM Agent    → 输出任务规划
架构 Agent  → 输出技术方案    [依赖 PM 输出]
实现 Agent  → 输出代码        [依赖 架构 输出]
审核 Agent  → 输出审核意见     [依赖 实现 输出]
验收 Agent  → 输出验收结果     [依赖 审核 输出]
Token = 5 × (prompt + completion) + 上下文传递
```

### 实际放大系数

| 阶段 | Agent | 输入 Token | 输出 Token | 累计 |
|------|-------|-----------|-----------|------|
| 1 | PM | 2,000 | 1,500 | 3,500 |
| 2 | 架构 | 3,500 | 3,000 | 10,000 |
| 3 | 实现 | 10,000 | 8,000 | 26,000 |
| 4 | 审核 | 26,000 | 2,000 | 28,000 |
| 5 | 验收 | 28,000 | 500 | 28,500 |

**结论：** Multi-Agent 模式的 Token 消耗是传统单次对话的 **10-30 倍**，但换来的是代码质量和正确性的显著提升。
