## 技术博客：VPS 部署的"救火"模式

### 问题

第一次部署 KITLAMT 独立站到香港 VPS，过程中发现大量问题。

### 典型故障链

1. **SSH 连不上** → 发现 IP 被防火墙拦截 → 修改防火墙规则
2. **Nginx 启动不了** → 端口 80 被占用 → 放行端口
3. **Node 服务 502** → nginx 代理配置不对 → 修改 `proxy_pass` 路径
4. **数据库连不上** → PostgreSQL 默认只监听 localhost

### 为什么会出这么多问题

因为是一台**新 VPS**，什么都没有配。换了五六个环境：
- 香港 VPS（CentOS/Ubuntu）
- SSH 密钥对认证
- Nginx + Node + PostgreSQL 组合

每一步都可能出问题，事实上每一步都出了问题。

### 教训

应该在部署前做一份**检查清单**：
- [ ] SSH 能连
- [ ] 基础软件已安装（nginx, node, psql）
- [ ] 防火墙规则已放行（80, 443, 3000）
- [ ] PostgreSQL 可远程连接
- [ ] 域名已解析到 IP
- [ ] 静态文件可直接访问

没有清单的后果就是——出了一个问题修一个问题，修完发现还有下一个，永远不知道全貌。

## 知识库

### 快速部署脚本框架

```powershell
# check_site.ps1 — 一键检查网站状态
$url = "http://$server_ip"
try {
    $resp = Invoke-WebRequest -Uri $url -TimeoutSec 10
    Write-Output "状态: $($resp.StatusCode)"
} catch {
    Write-Output "失败: $_"
}
```

### Nginx 502 排查步骤

```bash
# 1. 检查 Node 服务是否在运行
ps aux | grep node

# 2. 查看 nginx 错误日志
tail -f /var/log/nginx/error.log

# 3. 测试 nginx 配置
nginx -t

# 4. 重启
systemctl restart nginx
```

### SSH 快速修复

```powershell
# check_sshd.ps1 — 检查 SSH 服务状态
ssh $user@$ip "systemctl status sshd"
ssh $user@$ip "ss -tlnp | grep :22"
```

### 大文件分片传输

```bash
# 打包并分片（195MB 资源）
tar -cf assets.tar assets/
split -b 36M assets.tar assets_part_
cat assets_part_* > assets.tar  # 在目标端合并
```

## 复盘

**对的事：** 没有放弃 SSH 硬碰硬，而是一直在排查和修复。VPS 部署就是这样的——第一次总是最痛的。

**不对的事：** 没有提前做部署清单，导致每次修复都是"被动救火"。同一个问题（502）查了两次才找到根因。

**根本问题：** 新 VPS 的初始化应该自动化，而不是手工一步步来。下次应该写一个 `bootstrap.sh` 一键脚本。

---

## 今日收获

- 新 VPS 部署需要检查清单，不要凭感觉来
- 502 大概率是 nginx 代理路径或 Node 服务挂了
- 大文件传输要分片，单个超过 50MB 直接传容易中断
- 脚本救火一时爽，但真正的解决方案是一键初始化脚本
